# Глубокий скан безопасности модуля (Module Security Scanner)

Выполнить глубокий и изолированный анализ безопасности одного конкретного модуля или компонента системы (например, Auth-сервиса, Billing-модуля, File Uploader'а).
Этот скилл использует паттерн мульти-агентного сканирования для предотвращения "размытия фокуса".

> [!IMPORTANT]
> Вы обязаны делегировать проверку 3 специализированным суб-агентам. Не пытайтесь анализировать весь модуль одним промптом.

## Подготовка

Определите границы модуля, который вы анализируете (какие файлы, папки или классы в него входят).

## Запуск суб-агентов

Launch sub-agents using the `task` tool. Each sub-agent runs independently with its own prompt below.

Запустите 3 параллельных суб-агента со следующими строгими специализациями. Передайте им исходный код анализируемого модуля.

### Суб-агент 1: Эксперт по потоку управления (Control Flow & State Hacker)
**Цель:** Найти способы сломать логику состояний.
**Промпт для агента:**
"Ты - эксперт по логическим уязвимостям. Твоя задача изучить предоставленный модуль и найти способы нарушить его State Machine.
1. Найди Race Conditions (состояния гонки): что будет при параллельных запросах?
2. Найди способы пропустить обязательные шаги (например, оплатить товар, не добавляя его в корзину, или сменить пароль без ввода старого).
3. Проверь транзакционную целостность: что будет, если скрипт упадет посередине выполнения? Останутся ли данные в неконсистентном состоянии?
Верни ТОП-3 критические логические уязвимости с указанием места в коде и сценария атаки."

### Суб-агент 2: Эксперт по инъекциям и данным (Data Poisoning & Injection Expert)
**Цель:** Найти способы внедрить вредоносный payload.
**Промпт для агента:**
"Ты - эксперт по инъекциям. Твоя задача найти все точки, где модуль принимает внешние данные.
1. Проверь все параметры, заголовки, куки, которые парсит модуль.
2. Ищи SQLi (прямые или Second-Order), NoSQLi, XSS, Command Injection, SSRF (если модуль делает внешние запросы).
3. Проверь валидацию типов (что будет, если вместо строки передать массив, огромный JSON или null?).
4. Проверь Path Traversal и безопасность работы с файлами (если модуль загружает или читает файлы).
Верни ТОП-3 точки внедрения с указанием файла, вектора и payload'а, который может сломать систему."

### Суб-агент 3: Эксперт по доступу и утечкам (Access & Leakage Auditor)
**Цель:** Найти обходы авторизации и утечки чувствительных данных.
**Промпт для агента:**
"Ты - эксперт по авторизации и криптографии. Твоя задача проверить, кто и как получает доступ к модулю.
1. Проверь наличие IDOR (Broken Object Level Authorization). Проверяет ли модуль, что данные принадлежат тому, кто их запрашивает?
2. Проверь утечку секретов в логах, в ответах API (возврат хэшей паролей, ключей).
3. Проверь криптографию (слабые алгоритмы, хардкод секретов, слабая генерация рандома для токенов).
Верни ТОП-3 проблемы доступа или утечек с указанием файла и способа эксплуатации."

## Сведение результатов (Агрегация)

После возврата ответов от всех трех суб-агентов, выполните контрастный анализ:
1. Очистите результаты от дубликатов.
2. Отбросьте откровенные галлюцинации (если агент указал на несуществующий файл или библиотеку).
3. Сформируйте финальный **Отчет о безопасности модуля**:

```markdown
# Отчет по безопасности модуля: [Название Модуля]

## 🚨 Критические уязвимости (Блокируют релиз)
[Уязвимости уровня Critical с детальным описанием вектора и рекомендациями]

## ⚠️ Серьезные угрозы (Major/High)
[Уязвимости, требующие исправления в ближайшее время]

## ℹ️ Логические и архитектурные недочеты (Minor/Medium)
[Проблемы состояния, отсутствие транзакций, потенциальные гонки]
```

**Результат сохранить в**: `.opencode/context/module-security-report.md`
