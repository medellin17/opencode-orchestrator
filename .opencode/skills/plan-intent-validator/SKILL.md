# Проверка соответствия плана исходной задаче

The Conductor will provide the original task and the PLAN.md content. If they are provided as template variables, substitute them before analysis.

Сравнить итоговый план реализации (`PLAN.md`) с исходной задачей пользователя. Выявить расхождения по покрытию требований, scope creep и искажение интерпретации.

## Промпт

Исходная задача пользователя:
"{{task}}"

Итоговый план реализации:
{{PLAN.md}}

Выполнить 5-проходную проверку alignment плана с исходной задачей.

> [!IMPORTANT]
> Не писать код. Не дорабатывать план. Только анализ и вердикт.

---

## Проход 1. Извлечение intent

Из исходной задачи выделить:
1. **Главная цель** — одним предложением, что должно получиться в результате.
2. **Обязательные требования** — конкретные функции, поведение или артефакты, без которых задача не считается выполненной.
3. **Явные ограничения** — технологии, scope, сроки, объём, что явно исключено.
4. **Неявные ожидания** — разумные предположения о качестве, безопасности, тестах, документации, которые следуют из контекста задачи.

---

## Проход 2. Проверка полноты покрытия

Сравнить каждое обязательное требование из Прохода 1 с содержанием `PLAN.md`.

Для каждого требования указать:
- **Покрыто** — в какой секции плана и как конкретно.
- **Покрыто частично** — что именно упущено.
- **Не покрыто** — требование отсутствует в плане.

Определи недостающие или частично покрытые требования. Если таковых нет — явно напиши: «Все требования покрыты».

---

## Проход 3. Проверка на scope creep

Выявить элементы плана, которые:
- не запрашивались в исходной задаче;
- выходят за явные ограничения;
- добавляют функциональность, архитектурные слои или зависимости без прямой необходимости для цели.

Для каждого найденного элемента указать:
- Что добавлено.
- Почему это выходит за рамки задачи.
- Влияние на объём, сроки и сложность.

При отсутствии scope creep явно написать: "Scope creep не выявлен".

---

## Проход 4. Проверка интерпретации

Проверить, не переформулирована ли задача так, что решается другая проблема:
- Соответствует ли "Зачем?" в плане исходной боли пользователя?
- Соответствует ли "Что?" заявленному результату из задачи?
- Нет ли подмены цели (например, вместо "добавить уведомления" план решает "переделать всю систему событий")?

При отсутствии искажений явно написать: "Искажение интерпретации не выявлено".

---

## Проход 5. Проверка критериев готовности

Для каждой секции "Как проверить?" в плане:
- Проверяет ли критерий то, что описано в секции "Что?" этой же карточки?
- Можно ли однозначно доказать выполнение критерия (конкретные метрики, сценарии, edge cases)?
- Нет ли проверок, которые замыкаются на самих себя (например, "код написан" вместо "функция работает")?

Найти минимум 1 критерий, который не проверяет заявленное поведение. Определи недостающие или частично покрытые требования. Если таковых нет — явно напиши: «Все требования покрыты».

---

## Итоговый вердикт

На основе проходов 1–5 выдать один из двух вердиктов:

### `ALIGNED`
Использовать только если одновременно выполнены условия:
- Все обязательные требования покрыты.
- Scope creep не выявлен.
- Интерпретация задачи не искажена.
- Критерии готовности проверяют заявленное поведение.

### `NEEDS_REVISION`
Использовать при любом расхождении.

Вывести структурированный отчёт:

```
Вердикт: ALIGNED / NEEDS_REVISION

Полнота покрытия:
- [✓ / ~ / ✗] Требование: ... → Статус: ...

Scope creep:
- ... (или "Не выявлен")

Искажение интерпретации:
- ... (или "Не выявлено")

Проблемы с критериями готовности:
- ... (или "Не выявлены")

Рекомендация:
- Если ALIGNED: "План можно утверждать и передавать к реализации."
- Если NEEDS_REVISION: конкретный список, что передать в plan-refiner для доработки.
```

> [!WARNING]
> Не занижать severity ради положительного вердикта. Если есть хорошо обоснованное расхождение — обязательно выдать `NEEDS_REVISION`.
