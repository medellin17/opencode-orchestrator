# Доработка плана реализации

Обновить существующий план на основе списка подтверждённых замечаний и проблем,
полученных от ревьюера. Результат — исправленный `PLAN.md` в корне проекта.

## Структура плана (4 секции)

Каждый этап плана должен сохранять эту структуру:

1. **Зачем?** — цель этапа, какую проблему решает, почему именно так
2. **Что?** — конкретные файлы, функции, классы, которые будут изменены/созданы
3. **Как?** — последовательность действий, ключевые решения, зависимости
4. **Как проверить?** — конкретные проверки: тесты, ручные сценарии, метрики

## Процедура

1. **Исправление дефектов**: внести изменения в те разделы плана, к которым
   относятся замечания ревьюера. Полностью устранить указанные риски и нестыковки.
   Не просто добавить примечание — переписать затронутые секции.

2. **Сохранение структуры**: каждый исправленный этап должен сохранять 4-секционную
   структуру (Зачем? / Что? / Как? / Как проверить?). Не сокращать, не объединять секции.

3. **Локальность изменений**: корректировать только те этапы, которые требуют
   доработки согласно замечаниям. Не переписывать корректные этапы.

4. **Проверка новых рисков**: убедиться, что исправления не создают новых
   кросс-доменных или системных конфликтов. Проверить:
   - Не нарушены ли зависимости между этапами
   - Не появились ли противоречия с соседними этапами
   - Не сломана ли последовательность

5. **Полный вывод**: записать итоговый план **целиком** в `PLAN.md` (перезапись).
   Запрещены сокращения вида «остальное без изменений» или «...».

## Пример применения замечания

**Замечание ревьюера**: «В этапе 3 не указано, как обрабатывать ошибку БД при миграции.»

**Неправильно** (просто добавить строку):
```
Как проверить?: запустить тесты
Примечание: нужно обрабатывать ошибку БД
```

**Правильно** (переписать секцию Как?):
```
Как?:
1. Создать миграцию alembic
2. В upgrader() обернуть выполнение SQL в try/except
3. При ошибке — откатить транзакцию, записать в лог, вернуть 500
4. При успехе — обновить версию схемы в meta таблице
```

> Не писать код в процессе корректировки плана. Сосредоточиться исключительно
> на процедурных и логических шагах.

**Результат сохранить в**: `PLAN.md` (корень проекта, полная перезапись).
